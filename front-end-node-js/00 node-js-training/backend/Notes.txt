=> What is backend core component?
   A backend system is made of 4 core layers
   - Programming Language
   - Database
   - Runtime Environment
   - Framework
   
   1. Programming Language (JavaScript):
      A programming language is what you use to write logic and instructions for your application.

      In backend:
      JavaScript is used to:
      - Handle requests
      - Process data
      - Apply business logic
      avaScript is single-threaded and uses an event-driven, non-blocking model, which makes it efficient for handling multiple requests.

    2. Database (MongoDB):
       A database is used to store and manage data permanently.

       In backend:
       MongoDB stores:
       - Users
       - Products
       - Orders
       MongoDB is a NoSQL database that stores data in flexible, JSON-like documents instead of tables.
       - Fast for development
       - Easy to scale
       - Works well with JavaScript (same JSON format)

    3. Runtime Environment (Node.js):
       A runtime environment is where your code actually executes.
       Node.js uses the V8 engine and an event loop to handle asynchronous operations efficiently.

       In backend:
       Node.js allows JavaScript to run outside the browser.

       Why it matters:
       - Can handle many users at once
       - Non-blocking (doesn’t wait for one task to finish)

       Example:
       - Multiple users sending requests → Node handles them without freezing

    4. Framework (Express.js):
       A framework is a tool that simplifies development by providing structure and features.

       In backend:
       Express helps you:
       - Create routes
       - Handle requests/responses
       - Use middleware
       Without Express: You’d write everything manually (complex + slow)

    Interview ready ans:
    Backend development consists of a programming language like JavaScript, which defines the logic; a runtime environment 
    like Node.js, which executes the code; a framework like Express.js, which simplifies API development; and a database 
    like MongoDB, which stores and manages data.

    Example:
    When a user sends a request, it is handled by the Node.js server, routed using Express, processed using JavaScript 
    logic, and data is stored or retrieved from MongoDB before sending a response.
    This architecture is asynchronous and non-blocking, which makes it scalable for handling multiple users.

=> MongoDB Atlas (with proper explanation):
   1. MongoDB Atlas is a fully managed cloud database service.
      - You don’t install MongoDB on your system
      - You don’t manage servers

      MongoDB handles:
      - Hosting
      - Scaling
      - Backups
      - Security
   
      # Why companies prefer Atlas (important for interview):
        MongoDB Atlas is used because it removes the overhead of managing database infrastructure and provides features like 
        automatic scaling, high availability, and remote access.
     
      # Cluster (important term):
        When you create a cluster: You are actually creating,
        - A group of database servers (nodes)
        - That store and manage your data
        A cluster ensures data availability and scalability in production environments.

   2. Connection Process:
      Step 1: Create Cluster:
      - MongoDB allocates servers
      - Sets up database instance
      - Prepares it for connections

      Step 2: Click “Connect”:
      Atlas generates a connection string (URI)
      This string contains:
      - Protocol (mongodb+srv)
      - Username
      - Password
      - Cluster address
      - Database name

      Step 3: Connection URI (deep understanding):
      mongodb+srv://username:password@cluster0.mongodb.net/mydb
      Now break it like an interviewer:
      - mongodb+srv → protocol for connecting to MongoDB cluster
      - username:password → authentication
      - cluster0.mongodb.net → database server address
      - /mydb → database name

      The URI is a standardized connection string that allows the backend application to authenticate and communicate 
      with the MongoDB cluster over the internet.

   3. Why we don’t hardcode URI:
      We avoid hardcoding credentials because it exposes sensitive data, reduces flexibility across environments, and 
      violates secure coding practices.

   4. .env file (proper explanation):
      A .env file is used to store environment-specific configuration values
      What we store:
      - DB URI
      - API keys
      - JWT secrets
      - Port numbers

      Why it is important:
      It helps separate configuration from code, making the application more secure, maintainable, and 
      environment-independent.

   5. How Node.js uses .env:
      process.env.DB_URI
      process.env is an object that stores environment variables.
      In Node.js, environment variables are accessed using process.env, which allows secure and dynamic configuration of 
      the application.

   6. Why we keep .env file in .gitignore:
      .env is added to .gitignore to prevent sensitive credentials from being pushed to public repositories, 
      which could lead to security breaches.
   

   7. Explain how you connect backend to MongoDB(final Answer):
      First, we create a MongoDB Atlas cluster, which provides a cloud-hosted database. Then we obtain a connection URI 
      that includes authentication credentials and cluster details. For security, we store this URI in a .env file instead 
      of hardcoding it. In the Node.js application, we access this URI using process.env and use a library like Mongoose to 
      establish a connection between the backend server and the database.

=> What is ES Module?
   ES Modules are the standard JavaScript module system that allows us to organize code using import and export 
   statements. In Node.js, we enable ES Modules by setting 'type': 'module' in package.json, and it provides a cleaner 
   and more modern way to structure applications compared to CommonJS.

=> What is Mongoose?
   Mongoose is a library (ODM) used in Node.js to interact with MongoDB in a structured way.

   # What does ODM mean?
     ODM = Object Data Modeling
     It means:
     - You work with JavaScript objects
     - Instead of writing raw database queries

=> Why we NEED Mongoose:
   1. Schema (structure of data):
      You define how your data should look
      Example:
      name → string
      email → required

   2. Validation (very important):
      email: {
        type: String,
        required: true
      }
      Prevents wrong data from entering DB

   3. Easy queries
      Instead of complex queries:
      User.find({ email: "test@gmail.com" });
      Simple and readable

   4. Models (clean code structure):
      You create models like:
      - User
      - Post
      - Product

   # Final ans:
     Mongoose is an Object Data Modeling library for MongoDB in Node.js. It provides a schema-based structure to define 
     data models, supports validation, and simplifies database operations. It acts as a bridge between the application 
     and MongoDB, making data handling more organized and maintainable compared to using raw MongoDB queries.
